Overview

An Airflow-native bridge to pydantic-ai

The provider wraps pydantic-ai so you can call LLMs, route branches, analyze files, generate SQL, and run multi-turn agents from inside a dag.

  • Provider-agnostic: OpenAI, Anthropic, Bedrock, Vertex AI, Ollama, Azure
  • Structured outputs via Pydantic output_type
  • Agents with toolsets: SQL, MCP, DataFusion, Hook, Logging
  • Human-in-the-loop review ships as a FastAPI plugin
Who it is for

Teams that already orchestrate pipelines in Airflow and want LLM reasoning in the same execution model, not a separate agent runtime. One place to debug, secure, schedule, and observe.

Connections

Two conn_types configure the whole surface

pydanticai

Points every LLM operator and agent at a pydantic-ai model. Swap providers by editing one field.

{ "conn_type": "pydanticai", "extra": { "model": "openai:gpt-5-mini" } }

swap prefix: openai: / anthropic: / bedrock: / google-vertex: / ollama: / azure:

mcp

Points agents at a Model Context Protocol server. stdio launches a subprocess; http hits a remote endpoint.

{ "conn_type": "mcp", "extra": { "transport": "stdio", "command": "python", "args": ["server.py"] } }

http variant: set transport to "http" and host to a URL.

Both connections can live in .env as AIRFLOW_CONN_* JSON, or be created in the Airflow UI / secrets backend.
Operator 1 of 6

LLMOperator  /  @task.llm

Single-shot LLM call. Takes a prompt, returns a string, or a Pydantic model if output_type=... is set.

from airflow.providers.common.ai.operators.llm import LLMOperator
@task.llm( llm_conn_id="pydanticai_default", system_prompt="You are a safety officer.", output_type=SeverityReport, ) def summarize(text: str) -> str: return f"Summarize: {text}"
Key parameters
llm_conn_id
pydanticai connection
system_prompt
role and rules
output_type
Pydantic model class
prompt
templated user message
Use for

Summarization, classification, extraction. Single-step, deterministic IO shape, no tool use. Fastest and cheapest primitive.

pydanticai connection
output_type
optional Pydantic model
Use for

Invoice / PDF extraction, image triage, log → structured findings. Any task where the input is a blob, not a string.

Operator 3 of 6

LLMBranchOperator  /  @task.llm_branch

The LLM picks which downstream task_id to run. Inherits BranchPythonOperator semantics: returned task runs, siblings are skipped.

from airflow.providers.common.ai.operators.llm_branch import LLMBranchOperator
@task.llm_branch( llm_conn_id="pydanticai_default", system_prompt=( "Route to fast_lane, review, " "or quarantine based on hazard." ), ) def route(payload: str) -> str: return f"Route:\n{payload}" route(input()) >> [fast_lane(), review(), quarantine()]
How it decides
  • Candidate branches are the downstream task_ids
  • system_prompt defines routing rules in natural language
  • Model returns one task_id, validated against the set
Use for

Natural-language triage, escalation, dispatching where a rules engine would be brittle or hard to maintain.

Operator 4 of 6

LLMSQLQueryOperator  /  @task.llm_sql

Turns a natural-language question into a single SQL string for a target dialect, grounded on the real schema of the listed tables. Returns the SQL; you execute it downstream.

from airflow.providers.common.ai.operators.llm_sql import LLMSQLQueryOperator
@task.llm_sql( llm_conn_id="pydanticai_default", db_conn_id="warehouse", table_names=["orders", "customers"], dialect="postgres", ) def nl_to_sql(question: str) -> str: return question
Key parameters
db_conn_id
warehouse to introspect
table_names
tables to expose
dialect
postgres / snowflake / sqlite / ...
Use for

Natural-language dashboards for non-SQL users, pre-generating queries for human review, seeding data checks.

Operator 5 of 6

LLMSchemaCompareOperator

Introspects two or more databases, hands their schemas to the model, and returns a compatibility report: renames, type drift, breaking changes.

from airflow.providers.common.ai.operators.llm_schema_compare import LLMSchemaCompareOperator
compare = LLMSchemaCompareOperator( task_id="compare_schemas", prompt=( "Flag mismatches that would break " "CDC from primary to replica." ), llm_conn_id="pydanticai_default", db_conn_ids=["primary", "replica"], table_names=["orders"], context_strategy="full", )
Report shape
{ "compatible": false, "mismatches": [...], "suggested_actions": [...] }
Use for

CDC / replication readiness, migration gates, warehouse-to-warehouse compatibility checks.

Operator 6 of 6

AgentOperator  /  @task.agent

A multi-turn pydantic-ai agent wrapped as an Airflow task. Accepts a system prompt, an optional output_type, and a list of toolsets the model may call. One task, up to N LLM turns.

from airflow.providers.common.ai.operators.agent import AgentOperator
@task.agent( llm_conn_id="pydanticai_default", system_prompt="You are a planner.", output_type=MissionPlan, ) def plan(brief: dict) -> str: return f"Plan: {brief}"
Key parameters
prompt
user message (templated)
system_prompt
role and rules
toolsets
optional list of Toolset
output_type
optional Pydantic class
enable_hitl_review
gate result on human review
Why

Multi-turn reasoning with tool use. Every tool call is logged automatically.

Hook 1 of 2

PydanticAIHook

The low-level API every LLM operator is built on. Instantiate the hook, build an agent with create_agent(...), call run_sync(...). Your escape hatch when the operators do not fit.

from airflow.providers.common.ai.hooks.pydantic_ai import PydanticAIHook
@task def run_agent(text: str) -> dict: hook = PydanticAIHook( llm_conn_id="pydanticai_default", ) agent = hook.create_agent( instructions="Return a SeverityReport.", output_type=SeverityReport, ) return agent.run_sync(text).output.model_dump()
Why drop down
  • Custom streaming or token callbacks
  • Fine-grained agent lifecycle control
  • Patterns the operators have not surfaced yet
  • Composing multiple agents inside one task
Returns

A pydantic-ai Agent with all upstream features: deps, tool fns, streaming, usage.

Hook 2 of 2

MCPHook

Low-level client for a Model Context Protocol server. Reads an mcp connection, starts a session, lets you list or invoke MCP tools directly. Normally wrapped by MCPToolset.

from airflow.providers.common.ai.hooks.mcp import MCPHook
@task def list_mcp_tools() -> list[str]: hook = MCPHook( mcp_conn_id="mcp_default", ) with hook.get_session() as session: tools = session.list_tools() return [t.name for t in tools]
When to reach for it
  • Inspecting or smoke-testing an MCP server
  • Custom integrations outside the agent flow
  • Invoking a single MCP tool without spinning up an agent
Transports

Honors the connection's transport: stdio or http. Same API either way.

Toolset 1 of 5

HookToolset

Lifts any Airflow Hook into the agent. Every public method in allowed_methods becomes a callable tool. Reuses hooks you already ship, no tool rewriting.

from airflow.providers.common.ai.toolsets.hook import HookToolset
toolset = HookToolset( hook=FSHook( fs_conn_id="local_fs", ), allowed_methods=[ "list_files", "read_file", ], )
Inputs
hook
any instantiated Airflow Hook
allowed_methods
method-name allow-list
Why it is powerful
  • Works with any custom hook your team ships
  • Security by allow-list, not by convention
  • No pydantic-ai tool-function boilerplate

Toolset 4 of 5

MCPToolset

Plugs a full Model Context Protocol server into the agent. Handles transport, tool discovery, and call lifecycle. Reuse any MCP server your org already ships.

from airflow.providers.common.ai.toolsets.mcp import MCPToolset
toolset = MCPToolset( mcp_conn_id="mcp_default", ) # agent = AgentOperator( # ..., toolsets=[toolset])
Transports
stdio
local subprocess, good for dev
http
remote server, prod-grade
Why MCP here
  • Tools live server-side; Airflow does not hardcode them
  • Same agent code regardless of transport
  • Swap local dev for remote prod with a connection change
Toolset 5 of 5

LoggingToolset

A decorator for toolsets. Wrap any other toolset, and every tool call, result, and duration lands in the task log with ::group:: folding for the Airflow UI.

from airflow.providers.common.ai.toolsets.logging import LoggingToolset
toolset = LoggingToolset( wrapped=SQLToolset( db_conn_id="warehouse", allowed_tables=["orders"], max_rows=100, ), )
Why you want this on by default
  • Agent decisions become auditable in the log view
  • Correlates token usage with tool behavior
  • No observability code in the dag itself
  • Composes with any other toolset in the ecosystem
Cross-cutting feature

Structured outputs with Pydantic

Every operator that returns more than a string accepts output_type=<BaseModel>. The provider validates, serializes to XCom, and hands the next task a typed object.

class SeverityReport(BaseModel): severity: Literal["low", "medium", "high", "critical"] summary: str affected_systems: list[str] recommended_action: str @task.llm( output_type=SeverityReport, ..., )
Payoff
  • Downstream code is plain Python, not regex on a string
  • Model mistakes fail loud, at validation, not silently
  • Models reusable across dags and review surfaces
  • Deterministic branching on enum-typed fields
  • Works with LLMOperator, AgentOperator, LLMFileAnalysisOperator, and PydanticAIHook
Cross-cutting feature

Compose toolsets on one agent

Pass a list of toolsets to one agent. Each tool is auto-namespaced; the model chooses. Wrap any toolset in LoggingToolset to get a full trace.

agent = AgentOperator( task_id="cross_reference", llm_conn_id="pydanticai_default", toolsets=[ LoggingToolset( wrapped=SQLToolset( db_conn_id="warehouse", allowed_tables=["users"], ), ), HookToolset( hook=FSHook(...), allowed_methods=["get_path"], ), MCPToolset( mcp_conn_id="mcp_default", ), ], )
Why compose
  • One agent reasons across SQL, files, and MCP tools in a single task
  • Each toolset brings its own safety rails
  • Swap a toolset without touching agent code
The agent decides which tool to call at each turn. Model-provided arguments are validated against each tool's typed signature.
Plugin feature

Human-in-the-loop review

Set one flag and the agent pauses on its proposed output, waiting for a human to approve, reject, or request changes via the built-in /hitl-review FastAPI plugin on the Airflow API server.

agent = AgentOperator( task_id="risky_action", llm_conn_id="pydanticai_default", toolsets=[...], enable_hitl_review=True, max_hitl_iterations=3, hitl_timeout=timedelta(hours=1), hitl_poll_interval=10, )
